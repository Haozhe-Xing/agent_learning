## 顺手整理了一些高频面试题

既然聊到这里，干脆把我整理的一些高频面试题也分享出来，都是我在这份资料里找到答案的。

---

### 🔥 基础概念类

**Q：Agent 和普通 LLM 调用有什么本质区别？**

普通 LLM 调用是"一问一答"，输入 → 输出，是无状态的单次推理。Agent 的核心是**感知-思考-行动的循环**（Perceive-Think-Act Loop）：Agent 能感知环境状态，调用工具改变环境，再根据新状态继续推理，直到目标达成。本质区别在于：Agent 有**持续的目标**、**工具调用能力**和**状态记忆**。

---

**Q：ReAct 和 Chain-of-Thought 有什么区别？**

CoT 是纯思维链，只在 LLM 内部推理，不与外部交互。ReAct（Reason + Act）在思考的同时穿插行动：

```
Thought: 我需要查一下今天的天气
Action: search("北京今天天气")
Observation: 晴，25°C
Thought: 天气不错，可以推荐用户出行
Answer: 今天北京天气晴好，适合外出...
```

ReAct 的优势在于每一步 Observation 都能修正后续推理，避免"幻觉滚雪球"。

---

**Q：ReAct 和 Reflexion 的区别？**

- **ReAct**：执行过程中交替思考和行动，是"边做边想"
- **Reflexion**：任务结束后对整个轨迹进行反思，把经验总结写入长期记忆，下次执行类似任务时参考，是"做完复盘"

两者可以结合使用：用 ReAct 执行任务，用 Reflexion 积累经验。

---

### 🔥 工具调用类

**Q：Function Calling 的完整消息流程是什么？**

```
1. 用户消息 → LLM（附带 tools 定义）
2. LLM 返回 tool_call（包含 function name + arguments）
3. 客户端执行工具，拿到结果
4. 把结果以 role="tool" 的消息追加到对话
5. 再次调用 LLM
6. LLM 生成最终自然语言回复
```

关键字段：`tool_call_id`（用于匹配调用和结果）、`function.name`、`function.arguments`（JSON 字符串）。

---

**Q：如何设计一个好的 Tool Description？**

面试高频考点。好的工具描述需要包含：
- **功能说明**：这个工具做什么
- **适用场景**：什么时候该用它（比"不该用"更重要）
- **参数说明**：每个参数的含义、类型、取值范围
- **返回值说明**：返回什么格式的数据
- **示例**（可选但加分）

LLM 选工具完全依赖 description，描述模糊 = 工具被误用或不用。

---

### 🔥 记忆系统类

**Q：Agent 的记忆系统分哪几层？**

三层架构：
- **工作记忆（Working Memory）**：当前对话的 context window，容量有限，任务结束即消失
- **短期记忆（Short-term Memory）**：近期对话摘要、当前任务状态，通常存在外部存储（Redis/数据库）
- **长期记忆（Long-term Memory）**：用户偏好、历史经验、知识库，通过向量数据库实现语义检索

重要信息从工作记忆"下沉"到长期记忆，需要时通过语义检索"上浮"回来。

---

**Q：RAG 和 Fine-tuning 怎么选？**

| 场景 | 推荐方案 |
|------|---------|
| 知识频繁更新 | RAG（改数据库即可，不用重新训练） |
| 需要特定输出风格/格式 | Fine-tuning |
| 私有文档问答 | RAG |
| 改变模型行为模式 | Fine-tuning |
| 成本敏感 | RAG（无需训练成本） |

实际生产中两者经常结合：Fine-tuning 调整风格，RAG 补充知识。

---

### 🔥 多 Agent 类

**Q：MCP、A2A、ANP 三个协议分别解决什么问题？**

这是 2025 年之后的新考点，很多人答不上来：

- **MCP（Model Context Protocol）**：解决 LLM 调用外部工具的标准化问题。定义了工具描述、调用、返回的统一格式，让任何 LLM 都能用任何工具，不用为每个工具写适配代码。
- **A2A（Agent-to-Agent）**：解决 Agent 之间任务协作的问题。一个 Agent 如何把子任务委托给另一个 Agent，如何传递上下文，如何处理结果。
- **ANP（Agent Network Protocol）**：解决 Agent 服务发现的问题。在一个 Agent 网络里，如何找到"谁能做这件事"。

三层协议栈：ANP 负责发现 → A2A 负责协作 → MCP 负责工具调用。

---

**Q：多 Agent 系统中，Supervisor 模式和去中心化模式怎么选？**

- **Supervisor 模式**：有一个中心 Agent 负责任务分配和结果汇总，适合任务有明确依赖关系、需要统一协调的场景。缺点是 Supervisor 是单点瓶颈。
- **去中心化模式**：Agent 之间通过消息队列或共享黑板通信，适合任务相对独立、需要高并发的场景。缺点是协调复杂度高。

---

### 🔥 强化学习类（2025 新考点）

**Q：GRPO 和 PPO 的主要区别是什么？**

PPO 需要一个单独的 Critic 模型来估计状态价值，显存占用约为 Actor 模型的 2 倍。

GRPO（Group Relative Policy Optimization）去掉了 Critic，改为**组内相对优势**：对同一个问题采样多个输出，用组内奖励的均值和方差归一化来估计优势值。显存只需约 1.5 倍模型大小，训练更稳定，DeepSeek-R1 用的就是这个方法。

---

**Q：Context Engineering 和 Prompt Engineering 有什么区别？**

Prompt Engineering 关注"怎么说"——如何措辞让 LLM 给出更好的回答。

Context Engineering 关注"LLM 看到什么"——在有限的 context window 里，如何组织工具结果、记忆片段、历史对话、系统指令，让 LLM 在每一步推理时都有最有用的信息。

Agent 时代，context window 是稀缺资源，Context Engineering 比 Prompt Engineering 更重要。
